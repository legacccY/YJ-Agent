## 第 1 章：背景科普（不懂生物也能看懂）

要理解这个项目，先得懂几个词。我用打比方的方式讲。

### 1.1 癌症新抗原疫苗是什么

癌细胞会发生基因突变，突变会让细胞表面冒出一些**正常细胞没有的、奇怪的小蛋白碎片**。这些碎片就叫**新抗原（neoantigen）**——你可以理解成癌细胞身上独有的"通缉令照片"。

癌症疫苗的思路：把这些"通缉令照片"（也就是新抗原对应的**肽段 peptide**，一串氨基酸）做成疫苗打进人体，训练免疫系统的"警察"（**T 细胞**）去专门追杀带这种标记的癌细胞。

### 1.2 免疫原性（immunogenicity）= 这张通缉令好不好用

问题来了：不是每张"通缉令"都能让警察认真办案。有的肽段打进去，T 细胞强烈反应（疫苗有效）；有的肽段打进去，T 细胞无动于衷（疫苗白打）。

**一个肽段能不能、以及能多强地激起 T 细胞反应，就叫它的"免疫原性"。**

做疫苗最关键的一步，就是从一堆候选肽段里**挑出免疫原性强的**。一个病人可能有几百个突变肽段候选，但疫苗只能放进去几个、十几个，所以"挑得准"价值巨大。

### 1.3 现有工具的局限：只会做是非题

过去十几年，学界做了一堆软件来预测免疫原性。但绝大多数工具只输出一个**二分类**结果：这个肽"有/无"免疫原性（或者一个 0-1 的概率，但训练时是按"有/无"训的）。

袁老师的洞察：**临床上需要的是"排序"，不是"有无"**。如果工具能告诉你"这 100 个肽里，第 7 号反应最强、第 23 号次之……"，医生就能优先把最强的几个放进疫苗。这种"预测反应强弱程度"的能力，英文叫 **response magnitude regression（响应强度回归）**，是这个领域公认的**空白（unaddressed gap）**——几乎没人专门做。

这就是大项目的终极目标：做一个叫 **QuantImmune** 的、能"定量强弱"的新工具。而要立这个项目，先得搞清楚：现有工具到底行不行？数据够不够？这就是你这块 benchmark 的价值。

### 1.4 ELISpot = 真实的"反应强弱"实验数据

要评测工具准不准，得有"标准答案"。标准答案来自一种实验叫 **ELISpot**：把病人的免疫细胞和某个肽段放一起，数有多少个细胞被激活、冒出信号点（spots）。**点越多 = 反应越强**。这个点数叫 **SFC（Spot-Forming Cells）**，就是肽段免疫原性强弱的"真值（ground truth）"。

袁老师给了你两个 ELISpot 真实数据集（下面 DS1 / DS2），它们就是评测的"考卷标准答案"。

### 1.5 还要懂的几个配角词

- **HLA（人类白细胞抗原）**：相当于细胞表面的"展示架"，把肽段举起来给 T 细胞看。每个人的 HLA 型号不同（比如 `HLA-A*02:01`），同一个肽段配不同 HLA，结果可能完全不同。所以工具输入通常是"肽段 + HLA 型号"一对。
- **MT / WT**：MT = mutant（突变型，癌细胞的），WT = wild type（野生型，正常细胞的原版）。很多工具要对比这俩才能判断。
- **提呈（presentation）vs 免疫原性（immunogenicity）**：这是个大坑。"提呈"指 HLA 这个展示架能不能把肽举起来（第一步），"免疫原性"指举起来之后 T 细胞认不认（第二步）。**能被举起来 ≠ 一定能激起反应。** 有的工具（比如 HLAthena）只预测第一步提呈，不预测免疫原性——这个区别后面会反复出现。

## 第 2 章：你具体要交什么（任务定义）

袁老师给你的明确 deliverable（交付物）是：

**每个工具部署测试后，收集 4 类信息，最终做成 PPT：**
1. **输入数据的模板 / 格式**（这工具吃什么样的数据）
2. **运行参数设置**（有哪些可调的旋钮、各管什么）
3. **输出数据的格式及含义**（吐出来的结果怎么读）
4. **工具的简要介绍**（特点、优势）

这 4 类信息，每个工具都写进了 `TOOLS/<工具名>.md`，再汇总进 PPT。

**额外（你超额做的）**：把工具放在 ELISpot 数据上跑出分数，做横向 benchmark，算出每个工具到底准不准。这部分不是袁老师硬性要求，但极大增强了交付价值，也是给 QuantImmune 立项的关键证据。

## 第 4 章：两套考卷（数据集）

袁老师给的两个 ELISpot 真实数据集，是评测的标准答案：

- **DS1（Dataset1）**：82 行，全是 9mer（9 个氨基酸长的肽）。**全部是阳性**（都有免疫原性，SFC 16-677）。因为没有阴性样本，**算不了 AUC**（区分能力指标），但可以测"能不能排出强弱顺序"。
- **DS2（Dataset2）**：101 行，变长肽（15-29mer）。**有阳有阴**（90 阳 + 11 阴），可以算 AUC。是主力评测集。

> ⚠️ **DS2 的命门**：阴性样本只有 **11 个**，而且来自 **9 个病人**，前 2 个病人就占了 45% 的阴性。样本太少 → 统计上很多排名"不显著"，这个 caveat 贯穿所有结论。

还有一个关键的工程动作：把这两个数据集的肽段，按工具要求的格式**炸开**成一张主干大表（`master_backbone.csv`，34247 行 = DS1 的 325 + DS2 的 33922，因为变长肽要滑窗切成多个子肽）。所有工具的分数都贴回这张表，方便横向对比。

## 第 6 章：你遇到的所有困难（踩坑大全）

### 环境 / 网络坑
- **Windows 存不了带 `*` 的文件名** → 改 WSL2 部署。
- **国内连不上 Docker Hub** → 改 WSL mirrored 网络 + VPN 代理 + 删停服镜像源。

- ImmuneApp：TF1.15 依赖回溯死循环，要先单装 tensorflow 再装其余。

- NeoTImmuML：R 语言 Peptides 包 2.4.6 的 API 和脚本对不上（scale 返 list 要 `[[1]]`、autoCorrelation 签名变了），逐个修，最后 demo 核验 76/78 列精确吻合。

### IMPROVE 的"结构性做不到"（一个重要的判断）
用户问"特征能不能完全不降级"。深查后发现：IMPROVE 有个特征要 **RNA-seq 表达量**，而 ELISpot 数据**根本没有 RNA-seq**。这不是部署没配好，是**数据本身就缺这一块**——结构性不可能。所以正确做法是把"需要表达量"当成这个工具的"输入要求"信息写进 PPT（恰好是袁老师要的第 ① 类信息），而不是去过度工程硬补。这是一次"识别出该停手的地方"的好判断。

### 静默失败的调试方法论
HLAthena 那 3 个 bug，没有靠猜，全用**控制变量二分法**定位（纯 9mer OK / 8+9 混 FAIL → 锁定混长度问题）。这是项目里反复用的硬功夫：工具显示打架时先怀疑工具不是数据，存疑的量用确定性单值测量。

> 还有一次教训：HLAthena 收尾时，主线工具 IO 管道间歇失效，误把乱码读成"336 完整"。后来确权真实状态是只跑成 166/336。教训记进了项目：HPC probe 一律结果落地文件防 stdout 丢失。

## 第 8 章：怎么把十个工具拉到同一条起跑线（横评方法与依据）

这一章是整个 benchmark 最容易被外行忽略、却是评审最先盯的地方：**十个工具天生根本没法直接比，要先做一大堆"对齐"工作，才能让它们站在同一条起跑线上。** 不讲清这套方法，后面第 9 章那张 AUC 排名表就是空中楼阁。

### 8.1 为什么不能直接比（三个"天生不齐"）

把这十个工具想成十个考生，但他们连题都没法一起考——有三处天生不一样：

1. **吃的不一样（输入格式不齐）**：DeepImmuno 吃"肽段 + HLA 型号"两列；PredIG 还要蛋白序列；pTuneos 原本要全基因组测序的 VCF 文件；IMPROVE 吃的是先算好的一堆特征。同一个肽段，喂给不同工具要写成完全不同的格式。
2. **吐的不一样（输出尺度 / 粒度不齐）**：有的吐 0-1 的概率分，有的吐排名分；更麻烦的是**粒度**——一个变长肽（比如 20mer）要先滑窗切成一堆 9mer/10mer 子肽，每个子肽再配多个 HLA 型号，于是一个肽在工具眼里变成"几十个小分数"，而真值（ELISpot）只有"这一个肽一个数"。分数一堆、真值一个，对不上。
3. **测的标准不一样（评测口径不齐）**：每个工具的原论文用各自的考卷、各自的指标（有的报 AUC、有的报命中率），各说各准。

> 一句话：**工具自带的"我很准"都是在各自主场、各自口径下说的；要公平横评，必须把它们强行搬进同一个客场、同一套规则。**

### 8.2 拉齐的七步流水线（同一条起跑线怎么搭出来）

整套对齐在 `scripts/prepare_inputs.py`（输入）+ `scripts/merge_results.py`（合分）+ `analysis/` 指标脚本（评测）里实现。按顺序是：

**第一步 · 统一输入（同一份考题，翻译成每个考生能读的语言）**
用 `prepare_inputs.py` 把**同一批** DS1/DS2 肽段，分别转成每个工具要求的输入格式。三个关键归一化：① HLA 型号写法统一（`HLA-A*02:01` / `HLA-A0201` 各工具写法不同，全部归一）；② 变长肽**滑窗**切成工具能吃的等长子肽；③ 每一步都**逐字对照李紫晨用 PRIME 跑的参考输出文件验证**，确保翻译没翻错题意。题目内容一字不改，只换"语言"。

**第二步 · 让吃不了全套数据的工具也能上线（子模型 / 降级，但标清楚）**
有的工具原本要的数据 ELISpot 没有，硬比就不公平也不可能。处理办法是**诚实降级、标明出身**：pTuneos 只用它能吃纯肽段的子模型 Pre&RecNeo；IMPROVE 缺 RNA-seq 表达量那一列做 impute 降级；NeoTImmuML 官方权重拿不到，用自训版上线并标 ★。这样它们也能站上起跑线，但 PPT / 报告里都标清楚"它是靠哪条腿上来的"，不假装满血。

**第三步 · 所有分数贴回同一张大表**
把每个工具跑出来的分，全部贴回主干大表 `master_backbone.csv`（34247 行 = 子肽 × HLA）。一张表对齐到行，谁覆盖了谁没覆盖一目了然。

**第四步 · 统一聚合（最关键一步，把"粒度不齐"彻底拉平）**
这是解决 8.1 第 2 点的核心。不管工具内部把一个肽切成多少个子肽 × 多少个 HLA、吐出多少个小分数，**最后都按 `Peptide_ID` 压成"每个肽一个分"**，这样才能和"每个肽一个真值"对齐。聚合方式做了三种并行对照：**max（取最高子窗，主口径）** / mean（平均）/ top3mean（前三平均）。主口径锁定 `max`——直觉是"这个肽只要有任意一个子窗/HLA 强，就算这个肽有潜力"。

> 📌 **这一步不是我们拍脑袋，是 neoantigen 流水线的标准做法（有文献依据）**：取最强子窗/最强 allele = 业界叫的 **best-binder 聚合**。pVACseq 官方就是 `Best MT IC50 Score = Lowest IC50 binding affinity`，跨所有肽长 + 跨病人所有 HLA allele 取最强（原话 "chooses the single best binding value representative across different epitope lengths...per mutation between all independent HLA alleles"，`--top-score-metric=lowest` 是标准参数）；IEDB/Galaxy 教程的聚合函数直接叫 "Aggregator Function = Maximum"；NetMHCpan-4.1 扫蛋白时 "print only the strongest binding peptide"。**生物学依据**：一个肽只要能被病人**任一** HLA 强提呈就可能引发 T 细胞应答（MHC-I 限制性），所以"任一 allele 取最强"正是这个前提的直接操作化（MuPeXI / pVAC-Seq 都对病人 6 个 allele 取最佳）。我们的"分数越高 = 越强"和它们的"IC50 越低 / %Rank 越低 = 越强"一一对应。来源全部落在 `reference/BENCHMARK_METHODOLOGY.md`。

> ⚠️ **切割会不会把"可能有反应的那段"切掉？（评审必问，必须答清）** 不会被切掉，但切法确实影响结果，要分清两件事。
> - **为什么不会切掉**：我们的滑窗是**步长 1、穷举所有起点 × 所有长度 8-14mer 的重叠覆盖**（`prepare_inputs.py`：`for win in range(8,15): for pos0 in range(0, full_len-win+1)`），不是切成不重叠的块。一条长肽内所有可能的短表位（所有注册框）都被某个子窗覆盖到了；MHC-I 真正呈递的最小表位绝大多数是 8-11mer，落在 8-14 范围内 → 真表位一定在某个子窗里。配上 max 聚合（取最强子窗），等于"把长肽里最可能起反应的那段自动挑出来"——穷举切 + 取最强，正是为了堵住"漏掉反应区"这个洞。
> - **切法真正影响结果的两处**：① **各工具支持的肽长范围不同**（DeepImmuno 只吃 9-10mer、IMPROVE 8-12、PredIG/pTuneos 8-14）——若某肽真正起反应的表位是 11mer，DeepImmuno 结构上根本看不到它，这条肽对它天生吃亏。这不是我们切割的问题，是工具自身输入限制，已体现在覆盖表的 NaN / 长度范围标注里。② **max 聚合的生物假设**：穷举切假设"任何子窗都可能被呈递"，但真实抗原加工（蛋白酶切割）有偏好，预测最强的子窗未必真被切出来——这是所有 best-binder 方法（含 pVACseq）的共同盲点，我们靠并行报 mean/top3mean 侧面检查。
> - **DS1 不涉及切割**：DS1 全是 9mer 短肽，整条直接喂，不滑窗。
>
> 一句话：**"切掉反应区"被穷举重叠 + 步长 1 的切法堵死了；切法对结果的真实影响是"各工具长度范围差异"和"best-binder 忽略加工偏好"，两者都已诚实标注。**

**第五步 · 统一真值**
ELISpot 的 SFC（点数）是唯一标准答案。DS2 里每个 `Peptide_ID` 取第一行的 ELISpot 值（同一肽各行同值）。所有工具对的是同一份标准答案。

> 📌 **用 ELISpot SFC 当"反应强弱"的连续真值也有依据**：Beyond MHC binding（10.37349/ei.2023.00091）明确 "a quantitative value which reflects the strength of the immunogenicity was set" 并用它跑 Spearman；Ott 2017 Nature 用 ">55 SFC 判阳"；ELISpot 综述（PMC3360522）说它 "highly quantitative, can measure a broad range of magnitudes of response"。功能性 IFNγ ELISpot 是公认的免疫原性金标准真值，不是 binding 代理。

**第六步 · 统一阈值（同一把刀切阳性）**
"多少 SFC 算阳性"会影响 AUC，所以不挑对自己有利的，而是三档全报：**>0（n_pos=90）/ >10（n_pos=64）/ >median=22.67（n_pos=50）**，所有工具用同一组阈值切，谁都不能挑自己最好看的那档当主结果。

**第七步 · 统一指标（同一套尺子量所有人）**
三类指标、同一个函数算所有工具：**AUC-ROC**（能不能分开"有/无免疫原性"）+ **AUPRC**（类别极不平衡时比 AUC 更可信）+ **Spearman ρ**（能不能排出"强弱顺序"，直接关联 ELISpot 实测值）。

### 8.3 凭什么说这才算"同一条起跑线"（五条公平依据）

光把流程跑通还不够，下面五条是"这场比较站得住脚"的依据，也是评审最会攻的点：

1. **单口径锁死 apples-to-apples**：主对比表**只认 `max, >0` 一种口径**。绝不允许"每个工具在 9 种（聚合 × 阈值）组合里各挑自己最高的那个"——那叫 selection-on-max，挑出来的数字彼此口径不同、根本不可比（红队 `REDTEAM 🟠-3` 专门标了这条）。各工具最优值另列一张表，并白纸黑字写"彼此不可比、不作排名依据"。
2. **覆盖差异透明**：工具缺肽 / 算不出分（NaN）的，分母不齐就如实标出来，不偷偷按 101 算。比如 **PRIME 缺 1 个肽段（有效 n=100，MixMHCpred 不支持的罕见 HLA）**、**deepHLApan 缺 3 个肽段（n=98，阴性只剩 10 个）**——这些都写进覆盖表，提醒"它们的 AUC 和别人 n=101 不完全可比"。
3. **泄漏 / 同源诚实声明**：PRIME / ImmuneApp / deepHLApan 都用 IEDB 数据训练，可能和 ELISpot 测试集有重叠，重叠会让分数偏乐观——这条 caveat 明写。方法学上对标学界四层防泄漏规范（peptide + 4 位 HLA 去重 / 测试肽现于训练则剔除 / 按患者切分 / 量化 overlap，见 `reference/BENCHMARK_METHODOLOGY.md`）。
4. **不能混比的单独拎出去**：HLAthena 只预测"提呈"不预测"免疫原性"（第 1.5 节那个大坑），和另外 8 个免疫原性工具不是一个任务，所以**单列为 presentation proxy，绝不塞进 8 工具的 apples-to-apples 表**。这是守住"同一条起跑线 = 同一个任务"的底线。
5. **可复现校验**：旧 5 工具用新口径复跑，AUC 与第一批结果 `max|diff ≤ 0.004`（浮点精度内），口径对齐 PASS；pTuneos / PRIME 的子模型对官方 example 对账 `r=1.0`（跑出来和官方一字不差，防伪通过）。证明这套对齐脚本本身没把分算错。

### 8.4 这套方法的依据来自哪（不是自己拍的）

整套口径不是凭直觉造的，是先派 researcher 通查 2020-25 年学界已发表的 neoantigen benchmark（TESLA / IMPROVE / PredIG / Comprehensive analysis / Beyond MHC binding），对标它们的标准做法（见 `reference/BENCHMARK_METHODOLOGY.md`）：**AUPRC 优先于裸 AUC**（极不平衡下 AUC 会高估）、**用 Spearman 关联 ELISpot 实测值**衡量定量强弱（正是 QuantImmune 的差异化卖点）、**还原真实临床阳性率**、**ground truth 必须是功能性 T 细胞实验（ELISpot）而非 binding**。学界共识是"独立集 AUC 普遍只有 0.52-0.65"——这条让我们的低分结果有了行业参照，证明不是我们做错了，而是这件事本来就难。

### 8.5 把家底摊开：哪些有文献依据，哪些是我们自己定的

这是最该说明白的一点——**这套方法不是全有现成文献，也不是全靠我们拍脑袋，是两者的拼装。诚实区分清楚，评审才信。** 下面把整个项目用到的方法逐条标出身：

**A. 有文献依据的（带引用，来源在 `reference/BENCHMARK_METHODOLOGY.md`）**

| 方法 | 依据来源 |
|---|---|
| **max / best-binder 聚合**（取最强子窗 × 最强 allele 作肽级分） | pVACseq `Best MT IC50`（跨长度+跨 allele 取最低 IC50）· IEDB/Galaxy "Aggregator = Maximum" · NetMHCpan-4.1 "strongest binding peptide" · NetMHCpan-4.0 "lowest %rank across alleles" |
| **best-over-allele 的生物前提**（任一 HLA 强提呈即有意义） | MHC-I 限制性 · MuPeXI / pVAC-Seq 对病人 6 个 allele 取最佳 |
| **ELISpot SFC 当连续 magnitude 真值** | Beyond MHC binding（quantitative strength + Spearman）· Ott 2017 Nature（>55 SFC）· ELISpot 综述 PMC3360522（highly quantitative）· PGV001 |
| **AUPRC 优先于裸 AUC、用 Spearman 测定量、还原真实阳性率、防泄漏四层、功能性真值** | IMPROVE · PredIG · TESLA · Comprehensive analysis · Beyond MHC binding（已对标，详见方法学档） |

**B. 我们自己定的（工程 / 红队决策，无单一文献规定，诚实标注、不冒充依据）**

| 决策 | 我们为什么这么定 |
|---|---|
| **同时报 max / mean / top3mean 三种聚合做对照** | max 本身有据，但"额外跑两种平行对照看结论稳不稳"是我们加的稳健性自检（防 max 偶然偏高） |
| **主口径锁死 `max, >0` 单口径、禁 selection-on-max** | 是我们红队（`REDTEAM 🟠-3`）自己立的纪律：不准每个工具挑自己最高分的口径，否则彼此不可比 |
| **阈值取 >0 / >10 / >median 三档** | >0 是 ELISpot 阳性的自然定义；>10 / >median 是我们加的敏感性分析，没有哪篇文献规定必须是这三档 |
| **master_backbone 大表（34247 行）+ 滑窗切肽 + HLA 归一的具体实现** | 整套 `prepare_inputs.py` / `merge_results.py` 是我们自己写的工程（做法标准，代码自写、自验） |
| **子模型 / 降级处理**（pTuneos 只用 Pre&RecNeo 子模型 · IMPROVE 缺表达量 impute · NeoTImmuML 自训版 ★） | ELISpot 没有这些工具要的全套数据，我们做的诚实降级，并逐一标清"靠哪条腿上线" |
| **复现校验**（旧 5 工具复跑 `max\|diff ≤ 0.004`、子模型对官方 example `r=1.0`） | 我们自己加的防伪环节，证明对齐脚本没把分算错 |

> 一句话区分：**"怎么聚合、用什么真值、用什么指标"主体有文献撑腰；"具体跑几种对照、锁哪个主口径、切哪几档阈值、降级怎么处理"是我们的工程判断。** 凡是我们自己定的，文档 / PPT 里都标"我们的口径选择"，绝不包装成"学界标准"——这条诚实线和数字核 csv 是一类纪律。

> 小结：**第 9 章那张排名表能成立，全靠这一章把"输入 / 子模型 / 聚合 / 真值 / 阈值 / 指标"六处全部强行对齐到同一口径，再用五条依据守住公平；而每一处的出身（有据 / 自定）都摊开说清楚。** 横评的可信度不在分数本身，而在"大家是不是真站在同一条起跑线上、而且这条线是怎么画出来的讲得明白"。

## 第 9 章：成果 —— benchmark 到底测出了什么

这是项目最有价值的产出。用真实 ELISpot 数据测了 8 个免疫原性工具，结论很诚实，对袁老师的立项是硬证据。

### 8.1 核心数字（DS2，AUC 越高越能区分有无免疫原性，0.5 = 瞎猜）

> 数字全部核过 `metrics_ds2_8tools.csv`（max 聚合 / 阈值 >0），下面是各工具点估：

| 工具 | AUC | 排名 |
|---|---|---|
| pTuneos | **0.753** | 1 |
| PredIG | 0.661 | 2 |
| NeoTImmuML | 0.655 | 3 |
| IMPROVE | 0.621 | 4 |
| ImmuneApp | 0.589 | 5 |
| PRIME | 0.528 | 6 |
| DeepImmuno | 0.481 | 7 |
| deepHLApan | 0.419 | 8 |
| （HLAthena） | 0.509 | proxy 单列，近随机 |

### 8.2 三条诚实结论（这才是重点）

1. **现有工具判别力普遍弱**：最好的 pTuneos 也才 0.75，多数在 0.5-0.66 之间，离"可靠"很远。

2. **"最优"统计上不可区分**：因为 DS2 阴性只有 11 个，做 bootstrap 置信区间发现，pTuneos 的 AUC CI 是 [0.598, 0.888] 极宽，**pTuneos vs PredIG / NeoTImmuML 的差距跨 0（不显著）**。所以不能说"pTuneos 最强"，只能说"pTuneos 仅显著超过弱工具，对最近的竞品统计上分不开"。**连"无增量的新工具 ImmuneApp"都和 pTuneos 分不开** → n_neg=11 啥都分不开。这是红队抓出来、必须保守表述的关键。

3. **定量能力（排强弱）整体很弱**：用 Spearman 相关系数测"能不能排出 SFC 强弱顺序"，最强的也才 IMPROVE ρ≈0.32 / PredIG ρ≈0.28（弱相关），pTuneos ρ 不显著。**DS1（全阳）上，8/9 工具排序能力≈随机，没一个能排出强弱。**

### 8.3 最重要的一句话结论（对袁老师立项的硬证据）

> **现有工具是"分类器"不是"定量回归器"** —— 它们勉强能判断"有/无"（DS2 能排，因为有阴性），但**完全排不出"强弱程度"**（DS1 全阳排序≈随机）。
>
> 这正面证明了袁老师"做一个能定量强弱的工具（QuantImmune）"的动机是站得住的——这个能力市面上确实没有。

### 8.4 其他发现
- **组合最优是幻觉**：把 TOP3 工具组合点估 AUC 0.81 看着高，但配对 bootstrap ΔAUC CI 跨 0，不显著；盲目组合全部 8 个反而更差（被弱工具拖累）。
- **deepHLApan AUC 0.419 低于随机**不是 bug，是分数饱和（中位 0.993，全挤在高分区）。

## 第 10 章：QuantImmune 立项调研（命门 + 天花板）

benchmark 之外，你还为大项目要立的新工具 QuantImmune 做了大量立项情报。三个关键结论：

### 9.1 蓝海（好消息）
普查 2024-26 年 12+ 新方法 + 6 篇综述，**0 个工具做连续 magnitude 回归**。方向不撞车，是公认的空白。最强佐证：PredIG 有 low/inter/high 分级标签却 binarize 丢弃了，CNNeoPP 能分 weak/strong 却只吐二分 → **数据里有 magnitude 信号，全行业选择性丢弃**。

### 9.2 命门（坏消息）—— 连续标签数据严重稀缺
要训"定量"模型，得有大量"肽段 + 连续反应强度"的标签。你实测了最权威的公开库 IEDB（下了 1.3GB / 57 万行）和 CEDAR：
- **肿瘤子集功能性连续 magnitude 正例只有 ~87-104 个唯一肽**（判据要 ≥10³，差了约 10 倍）。
- 连续值系统性稀疏，大头是 binding/IC50（提呈相关）不是免疫原性强度。
- TESLA 也只有 binary（37 正例，单个 consortium）。
- **裁定：连续 magnitude 回归的公开数据地基 FAIL（三源交叉确认）**。

这是个**拍板点**：连续回归地基不成立，退守方向有三条（序数分级 / 响应频率回归 / 自补 ELISpot 产连续标签），需要袁老师/徐伊琳定。

### 9.3 理论天花板
即使数据够，理论分析也指出回报封顶：因为"反应强弱"很大程度取决于病人特异的 precursor frequency（前体 T 细胞频率）和 TCR-seq 信息，这些在肽序列里根本没有 → ρ 上限约 0.4-0.6。所以 headline 不能吹"颠覆性"，应该押"临床 top-K 排序增量"这种可观测的小目标。

### 9.4 skeptic 的最强建议
红队 agent 给的建议很关键：**直接做"现有工具定量能力 benchmark"论文，不做新工具**——承重点最少（证据在手）、不会死于数据稀缺、符合"benchmark 族全活"的历史规律。这条路也最稳。最终共识：D（benchmark 主路）+ A/B 当 baseline + C 当金标准评测，claim 形状从"大胆新工具"降到"benchmark/empirical"。
